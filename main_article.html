왜 Git 인가 ?

시작하기전에 git는 svn과 달리 commit->push 라는 한단계를 더 거치게 된다. 이 부분 때문에 많은 분들이 불편하다고 생각하는 경우가 있는데 오히려 이 과정으로 인해 커밋을 자유자재로 다룰 수가 있으며 속도를 보장 하고 설령 네트워크가 되지 않는 상황에서도 얼마든지 가능하다.

  1. 속도가 빠르다.
    기존에 사용되던 형상관리 ( 대표적으로 SVN) 등은 기본적인 명령어가 네트워크상의 속도에 영향을 받으므로 네트워크 환경이 상당히 중요하며 다른 네트워크에 있는 컴퓨터와의 연관이 크게 작용한다.
        그러나 GIT는 모든 히스토리가 로컬디스크에 저장되므로 상대적으로 빠르게 동작한다
  2. Commit 로그 관리가 용이하다.
    여기서 잠깐!
          커밋 로그를 관리하는 회사가 몇군데나 될까요? 커밋 로그는 사실상 히스토리 판단에 매우 중요함에도 대부분 회사가 커밋로그를 관리하지 않습니다. 일정의 이유도 있을것이며 상호 협의가 되어있지 않는 곳도 많을 것입니다. 이 발표를 계기로 커밋로그에 대한 여러가지 정책들도 상의해보시면 좋을 것 같습니다.
          저희 팀 같은 경우 몇가지 예를 들어보면 커밋은 무조건 ISSUE 번호 단위로 만들어지며 그 이슈와 관련 없는 파일은 제외 후 따로 커밋을 진행합니다. 또한 하나의 이슈에 대해 여러커밋을 하는경우는 극히 드물며 대부분 커밋을 합쳐서 Pull Request 요청을 진행합니다. (여러커밋인 경우는 Refactoring 로그를 남기기 위해서)
        
    svn의 경우 실수로 한 커밋을 고치기가 상당히 까다로우며 커밋을 합치거나 취소하기도 상당히 어려운 부분이 많다. git는 이러한 부분에 대해서 완벽하게 지원하고 있다. (뒤에서 설명)
  3. 브랜치를 다루는 데 있어 편리하다.
    git는 수십,수백개의 브랜치를 만드는것을 두려워하지 않아도 된다. 빠른 속도로 인한 (로컬에 모든정보를 저장하기 때문에) 쉽게 전환이 가능하다. 예를들어 현재 커다란 작업을 하는중에 특정 브랜치를 생성하여 작업을 진행하고 있는 와중에 다른 기능에서 오류를 패치해야한다. 그럴 경우 현재 작업은 그대로 납둔 상태에서 hotfix 브랜치를 하나 생성해서 작업해서 반영하면 된다. 그럴 경우 작업 중이던 브랜치 소스에는 영향이 없이 작업이 가능하다.
  4. git를 아름답게 쓸 수 있는 도구들이 있다.
    `gitflow`는 이상적인 브랜치모델링에 있어 아주 근사한 대답을 해주는 도구이다. (뒤에서 설명) 구글에서 안드로이드 개발에 사용한다고 하여 유명해진 `gerrit`이라는 도구는 누구나 머지를 할 수 있다는 git의 단점을 보완함과 동시에 코드리뷰를 진행토록 함으로써 코드에 대한 신뢰도를 높인다는데 그 의의가 있다.
  5. 오픈소스에 많이 사용되고 있다.
    오픈소스의 메카이자 성지인 github는 주 저장소로 git를 사용하고 있고 참여를 위해서는 git를 어느정도는 이해하고 있어야 적극적으로 참여가 가능하다.
가상 시나리오로 git명령어 보기 => 시연

  1. 원격 저장소에서 프로젝트를 받아서 브랜치를 생성하여 작업후 합치는 데 커밋을 정리하려는 상태. (커밋을 잘못해서 나눠서 커밋이 된 상황)

    # 외부에서 받아옴
  git clone https://localhost/git/test.git
  # 브랜치를 생성함
  cd test
  git checkout -b develop
  # 작업이 완료되고 커밋을 진행한다고 가정함
  git add .
  git commit -m "1번 커밋"
  git commit -m "2번 커밋"
  git commit -m "3번 커밋"

    이러한 상황에서 커밋 3개가 사실은 하나의 커밋으로 묶여야 하는 상황임을 알게 되어서 3개의 커밋을 합치겠다.

    # 최근 3개의 커밋을 합침
  git rebase -i HEAD&;3

    위의 명령어를 실행하면 다음과 같은 메세지가 뜬다.

    pick e4d311a 1번커밋
  pick vf30ds1 2번커밋
  pick 43fd1ad 3번커밋

    이 메세지를 다음과 같이 변경한다.

    pick e4d311a 1번커밋
  squash vf30ds1 2번커밋
  squash 43fd1ad 3번커밋

    저장을 하면 커밋메세지를 수정하는 창이 뜨고 입력하고 저장하면 해당 커밋 3개가 합쳐진 새로운 커밋이 만들어진다.

    # master 브랜치에 반영되어 있는 내용을 가져옴
  git rebase master
  # master 브랜치로 변경
  git checkout master
  # develop 브랜치에 변경사항을 master에 반영
  git rebase develop
  # 외부저장소로 push
  git push origin master

    여기서 잠깐 rebase 와 merge의 차이점을 알아 보도록 하겠다.
    이 내용만으로는 충분치가 않으며 시간관계상 간략하게만 설명하겠다.
    rebase : 이력재정렬 (history를 합친다.)
    merge  : 변경된 부분만 합친다.

  2. Rebase와 Merge확인을 위한 시나리오

    # 동일하게 진행되어질 부분
  touch 1
  git add .
  git commit -m "1번커밋-master"
  touch 2
  git add .
  git commit -m "2번커밋-master"
  git checkout develop
  touch 3
  git add .
  git commit -m "3번커밋-develop"
  touch 4
  git add .
  git commit -m "4번커밋-develop"
  git checkout master
  touch 5
  git add .
  git commit -m "5번커밋-master"

    merge를 진행하고 그 결과

    # merge
  git merge develop

    rebase를 진행하과 결과

    # rebase
  git checkout develop
  git rebase master
  git checkout master
  git rebase develop

    대부분 충돌나는 경우를 제외하거나 이미 push 되어 해당 커밋아이디를 사용하고 있는 경우가 아니라면 (rebase는 이력을 재정렬하기 때문에 커밋아이디가 바뀐다) 대부분 rebase를 선호한다. git의로그는 쫙 펴지는 로그가 깔끔하고 보기 좋기 때문이기도하다.

  3. 현재 작업 중인 브랜치에 아직 커밋을 할 수 없는 상황에서 다른 브랜치를 생성하여 급히 오류를 처리해야 하는 시나리오 (git는 stage영역에 반영되지 않은 데이터가 있는 경우 브랜치 변경이 되지 않는다)

    # 현재 브랜치에서 반영하지 않은 데이터 임시 저장
  git stash
  # 브랜치를 생성해서 작업함
  git checkout fix/issue-23
  # 반영되었다고 가정함 원래 작업하던 브랜치로...
  git checkout develop
  # 작업하던 stage 영역 원복
  git stash apply --index (이름)
  4. 주로 github프로젝트 참여시 발생하는 시나리오(Fork / Pull Request)
    # 원본 저장소에서 내 저장소로 Fork 한 상태 에서 내 저장소에 있는 것 clone
        git clone https://localhost/git/test/test.git
        # 원본 저장소 추가 git remote add [저장소이름] [주소]
        git remote add changhwa https://localhost/git/mate/mate.git
        # 브랜치 생성 하여 작업
        git checkout feature/more
        # 작업완료
        git add .
        git commit -m "feature more : issue end"
        # 원본 저장소에 변화가 있었는지 여부 확인
        git fetch changhwa
        # 변화가 있었을 경우 git rebase [저장소이름] [브랜치명]
        git rebase changhwa develop
        # Push 후 Pull Request 요청함
        git push -u origin feature/more

해당 이미지는 Vincent Driessen 님의 A successful Git branching model 에서 니용 하였습니다.

git-svn 연동

앞서 설명처럼 git는 상당히 즐거움을 주며 자유롭게 브랜치를 다루면서 개발도 좀 더 용이하게 할 수 있다는 장점을 줍니다. 그렇지만 아직 우리나라 많은 기업은 svn을 사용하고 있고 이에 따라 어쩔수 없이 svn을 택할 수 밖에 없는 경우도 종종 발생합니다. 그런 분들을 위한 챕터입니다.

git-scm.com/book/ko/Git으로-이전하기-Git과-Subversion 중 일부 내용입니다.
Git이 자랑하는 또 하나의 기능은 git svn이라는 양방향 Subversion 지원 도구이다. Git을 Subversion 클라이언트로 사용할 수 있기 때문에 로컬에서는 Git의 기능을 활용하고 Push 할 때는 Subversion 서버에 Push한다. 로컬 브랜치와 Merge, Staging Area, Rebase, Cherry-pick 등의 Git 기능을 충분히 사용할 수 있다. 같이 일하는 동료는 빛 한줄기 없는 선사시대 동굴에서 일하겠지만 말이다. git svn은 기업에서 git을 사용할 수 있도록 돕는 출발점이다. 우리가 Git을 도입하기 위해 기업내에서 노력하는 동안 동료가 효율적으로 환경을 바꿀 수 있도록 도움을 줄 수 잇다. Subversion 지원 도구는 우리를 DVCS 세상으로 인도하는 붉은 알약과 같은 것이다.

  1. SVN 프로젝트 받아오기
    -T, -b, -t 옵션은 trunk, branches, tags폴더를 git에게 알려주는 역할을 한다. 표준일경우는 -s 하면된다.
    모든 커밋을 가지고 오기때문에 저장소 크기에 따라서 상당한 시간이 소비 될 수 가 있다
    git svn clone http://localhost/svn/test/
  2. 작업 후 반영하기
    git commit -am "Add commit message"
  3. svn 서버에 반영하기
    git svn dcommit
  4. svn 서버에 변경된 사항 받아오기
    git svn rebase
    이 외에도 많은 명령어가 있지만 시간관계상 생략하고 git-scm 를 참고하면 좋겠다.
